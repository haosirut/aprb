# Сектор 5: Дуэль

Рабочее название для публикации в RuStore: **«Сектор 5х5: Дуэль»**.  
Логика названия: игра про дуэль на сетке, где поле игрока — сектор 5×5, и ключевое напряжение строится вокруг контроля клеток, разведки и блокировок. Коротко, звучно, легко ищется по словам «сектор» и «дуэль».

---

## 1) Концепция и цель проекта

Это кроссплатформенная мобильная пошаговая игра (MVP — Android), в которой два игрока сражаются на сетке 5×10 (вертикальная ориентация). Поле разделено на два сектора 5×5: сектор соперника сверху, сектор игрока снизу (на своём телефоне игрок **всегда снизу**).

Игровая механика строится на трёх вещах:

1) **ЮнитА** — боевые юниты, которые двигаются и стреляют.  
2) **ЮнитР** — разведка и блокировка клеток соперника (работают с туманом войны).  
3) **Клетки имеют прочность (HP)** и могут быть уничтожены целевой стрельбой по пустой клетке, чтобы ограничить перемещения и “посадочные места”.

Побеждает тот, кто уничтожит все ЮнитА соперника.

---

## 2) Варианты игры (режимы)

### 2.1 “Играть” (PvP)
Матч против живого соперника через серверный матчмейкинг и рейтинг.

### 2.2 “Тренироваться” (PvE)
Матч против компьютера, полностью локально (без сервера).  
Сложность на текущем этапе одна — самая простая.

---

## 3) Термины

**ЮнитА**  
Боевой юнит. Имеет HP, занимает клетку своего сектора. За раунд может выполнить ровно одно действие: перемещение или атака.

**ЮнитР**  
Разведывательный токен. Назначается на клетки сектора соперника в фазе ЮнитР. Может давать разведку или блокировку.

**Клетка**  
Элемент сетки. Имеет `cellHP` (прочность) и состояние разрушенности.

**Пакет действия**  
Группа выбранных ЮнитА из одной исходной клетки, которым назначается одно действие одной командой (например, атаковать одну цель).  
Важно: если игрок разбивает ЮнитА на несколько пакетов атаки, итоговый урон может быть меньше, чем при одной общей атаке.

**Разведанная клетка**  
Клетка соперника, которую игрок открыл ЮнитР. Разведка сохраняется навсегда.

**Туман войны**  
Неразведанные клетки соперника (скрыты). На них можно назначать ЮнитР “вслепую”.

---

## 4) Игровое поле

### 4.1 Геометрия
Глобальная сетка: **5×10**, вертикальная ориентация.

Сектора:
- Сектор соперника: 5×5 сверху.
- Сектор игрока: 5×5 снизу.

ЮнитА никогда не переходят между секторами.

### 4.2 Границы и “через край”
Перемещение “через край” сектора запрещено. Любые вычисления соседей для перемещения учитывают границы 5×5.

---

## 5) Сущности и параметры

### 5.1 ЮнитА
- На старте матча у игрока 5 ЮнитА.
- HP каждого ЮнитА на старте: `5`.
- ЮнитА — отдельные сущности: `unitAId`, `hp`, `cell`.
- В одной клетке может стоять от 0 до 5 ЮнитА. Переполнение невозможно, так как всего ЮнитА у игрока 5.

Отображение нескольких ЮнитА в клетке:
- Внутри клетки визуально показывается “шкала из 5 слотов” и/или разбиение на прямоугольники по количеству ЮнитА в клетке.
- У каждого ЮнитА свой HP.

Правило распределения входящего урона по ЮнитА в клетке:
- Урон сначала идёт в ЮнитА с наибольшим HP.
- При равенстве HP первым выбирается ЮнитА с меньшим `unitAId`.
- Остаток урона переносится на следующего ЮнитА по тем же правилам.

### 5.2 Клетки и их HP
- Каждая клетка имеет прочность `cellHP`.
- Стартовое значение: `cellHP = 10`.
- Клетка считается уничтоженной, если `cellHP <= 0`. Уничтоженная клетка:
  - получает красный контур;
  - становится недоступной для перемещения (`cellHP > 0` — обязательное условие входа);
  - на неё нельзя назначать ЮнитР.

Важно: клетка получает урон только при целевой атаке по пустой клетке (см. раздел “Урон и атака”).

### 5.3 ЮнитР
Основная игра (раунды 0–7): каждый раунд у игрока **4 ЮнитР**. Игрок может не использовать их вовсе.

Ограничения:
- В одну клетку можно назначить максимум 2 ЮнитР.
- Нельзя назначать ЮнитР на уничтоженные клетки (`cellHP <= 0`).
- Можно назначать ЮнитР в туман войны координатно “вслепую”.

Турбо-битва: у игрока **1 ЮнитР**, который автоматически создаёт блокировку выбранной клетки (и одновременно даёт разведку).

## 6) Время, раунды и фазы

### 6.1 Общий таймер на раунд
В каждом раунде используется **один общий таймер**, который покрывает обе фазы (фаза действий и фаза ЮнитР). Игрок сам распределяет время между фазами.

Таймеры:
- Раунд 0: 20 секунд
- Раунд 1: 50 секунд
- Раунд 2: 45 секунд
- Раунд 3: 40 секунд
- Раунд 4: 35 секунд
- Раунд 5: 30 секунд
- Раунд 6: 25 секунд
- Раунд 7: 20 секунд
- Турбо-битва: 15 секунд

Если время истекло:
- всё, что игрок успел назначить, остаётся;
- действия, которые игрок не успел назначить (для конкретных ЮнитА/ЮнитР), не происходят;
- не делается массовый “no-op всем”, это касается только тех, кому не назначили действие.

### 6.2 Порядок фаз (UI)
Раунд состоит из двух фаз:
1) Фаза действий: перемещение/атака.
2) Фаза ЮнитР: разведка/блокировка.

Кнопка “стрелка”:
- переводит игрока из фазы действий в фазу ЮнитР;
- затем из фазы ЮнитР завершает ход игрока в текущем раунде.

Стрелка используется только как переход фазы, а не как подтверждение команд: назначение команд происходит сразу при действии.

### 6.3 Запрет отмены действий
После назначения действия конкретному ЮнитА отменить или заменить действие нельзя.

Сброс выбора до назначения возможен (см. управление), но после назначения — действие закрепляется.

### 6.4 Один ЮнитА — одно действие
В рамках одного раунда каждому ЮнитА можно назначить ровно одно действие:
- либо перемещение,
- либо атака.

Комбинация “перемещение + атака” для одного ЮнитА в одном раунде запрещена.

---

## 7) Серверный резолв (авторитетная симуляция)

Для PvP сервер является источником истины и считает резолв детерминированно. Порядок резолва после завершения хода обоими игроками (или по таймеру):

1) Перемещения ЮнитА.
2) Атаки по клеткам:
   - сначала суммирование урона по целевым клеткам;
   - затем нанесение урона по правилам “ЮнитА или клетка”.
3) Применение ЮнитР:
   - разведка и блокировка;
   - блокировка действует на следующий раунд.

Раунд 0:
- ЮнитР, поставленные в раунде 0, применяются сразу после завершения 0 раунда, чтобы на старте раунда 1 уже было раскрытие/блокировка.

---

## 8) Перемещение ЮнитА

### 8.1 Радиус и направления
ЮнитА может переместиться на 1 клетку в одном из 8 направлений: вверх, вниз, влево, вправо и по диагоналям.

### 8.2 Условия допустимости перемещения
Перемещение возможно только в клетку, которая:
- находится в радиусе 1 (8 направлений);
- не выходит за границы сектора 5×5;
- имеет `cellHP > 0`;
- не заблокирована для входа (если блокировка активна в этом раунде).

---

## 9) Атака и урон

### 9.1 Цель атаки — клетка
Игрок выбирает целевую клетку соперника. Урон наносится либо ЮнитА в этой клетке, либо самой клетке — в зависимости от того, занята ли клетка ЮнитА после выполнения перемещений.

Игрок не “целиком целится в ЮнитА”. Он целится в клетку.  
Попал по ЮнитА или по клетке определяется после перемещений.

### 9.2 Урон пакета атаки
Если игрок сформировал пакет атаки из N выбранных ЮнитА (N от 1 до 5) из одной исходной клетки по одной целевой клетке, урон пакета равен:

`D(N) = 2N − 1`

В турбо-битве урон умножается на 3:

`D_turbo(N) = 3 * (2N − 1)`

### 9.3 Суммирование урона по клетке
В одном резолве по одной целевой клетке может прилететь несколько пакетов атаки. Пакеты могут быть сформированы:
- из разных исходных клеток;
- из одной исходной клетки, если игрок ошибочно разделил ЮнитА на несколько пакетов.

Правило резолва атак:
- сначала для каждой целевой клетки суммируется общий урон `U_total` от всех пакетов;
- затем этот суммарный урон применяется к содержимому клетки.

### 9.4 Куда идёт урон: ЮнитА или клетка
Если в момент нанесения урона (после перемещений) в целевой клетке есть хотя бы один ЮнитА соперника:
- весь суммарный урон `U_total` уходит только в ЮнитА;
- клетка не получает урон, даже если урон “уходит в минус” после уничтожения всех ЮнитА.

Если целевая клетка пустая:
- весь суммарный урон `U_total` наносится по `cellHP`.

---

## 10) ЮнитР: разведка и блокировка

### 10.1 Назначение и ограничения
Основная игра:
- 4 ЮнитР в раунд.
- Можно использовать 0.
- Не больше 2 ЮнитР на одну клетку.
- Не назначать на уничтоженные клетки.

Назначение в туман войны разрешено: игрок выбирает координаты сетки.

### 10.2 Эффекты
Если на клетку назначен 1 ЮнитР:
- клетка становится разведанной для игрока (после применения ЮнитР на шаге 3 резолва).

Если на клетку назначены 2 ЮнитР:
- клетка становится разведанной и получает блокировку на 1 раунд;
- визуальный цвет контура у цели — оранжевый.

### 10.3 Длительность и обновление блокировки
- Блокировка действует 1 раунд (следующий после применения).
- Если в клетку снова назначили 2 ЮнитР, блокировка обновляется ещё на 1 раунд.

### 10.4 Ограничения блокировки
В раунд, когда блокировка активна:
- запрещён вход в клетку;
- запрещён выход из клетки.

---

## 11) Управление в фазе действий (тапы и пакеты)

### 11.1 Тап по своей клетке с ЮнитА
При тапе по своей клетке, где есть ЮнитА, происходит сразу:
- подсветка клеток, доступных для перемещения (белое свечение контура);
- подсветка клеток соперника, доступных для атаки (зелёный контур);
- в нижней панели 1×5 появляются ЮнитА этой клетки, слева направо, с отображением HP.

### 11.2 Быстрое действие “всеми ЮнитА клетки”
Если игрок после тапа по своей клетке сразу тапает:
- по целевой клетке перемещения — перемещаются все ЮнитА из исходной клетки;
- по целевой клетке соперника — атакуют все ЮнитА исходной клетки одним пакетом.

### 11.3 Частичный выбор ЮнитА для действия
Игрок может выбрать часть ЮнитА:
- после тапа по исходной клетке, он тапает по конкретным ЮнитА в нижней панели (накапливая выбор);
- затем тапает по целевой клетке (перемещение или атака) и назначает действие только выбранным ЮнитА.

После назначения действия выбранным ЮнитА их нельзя переиспользовать в этом раунде; они отображаются неактивными (серые) в панели.

### 11.4 Пример ошибки “разбил пакет атаки”
Если в клетке 4 ЮнитА и игрок:
- сначала выбрал 2 и атаковал цель (урон D(2));
- затем выбрал оставшиеся 2 и снова атаковал ту же цель (ещё раз D(2));
то итоговый урон будет D(2)+D(2), а не D(4).  
Это осознанная “ловушка” на внимательность.

### 11.5 Сброс выбора до назначения
Пока действие не назначено (ещё не тапнули цель):
- повторный тап по той же исходной клетке сбрасывает текущий выбор ЮнитА (контекст выбора).

После назначения действия сброс/отмена невозможны.

---

## 12) Логика фазы ЮнитР (тапы)
В фазе ЮнитР:
- нижняя панель 1×5 показывает ЮнитР как токены (4 в основной игре, 1 в турбо);
- игрок назначает ЮнитР на клетки соперника:
  - один ЮнитР на клетку даёт разведку;
  - два ЮнитР на клетку дают блокировку и разведку;
- назначать можно и в туман войны, и на разведанные клетки;
- на уничтоженные клетки назначать нельзя.

---

## 13) Визуальные состояния клеток (контур, заливка чёрная)

Внутренняя заливка клетки всегда чёрная. Состояние задаётся контуром.

На поле соперника:
- тёмно-серый контур: туман войны (не разведана);
- белый контур: разведана ранее;
- жёлтый контур: разведана в этот ход (показывается только в момент результата/анимации после резолва, затем становится белой в следующем раунде);
- оранжевый контур: блокировка ЮнитР (и одновременно “разведана в этот ход”);
- красный контур: уничтожена (cellHP <= 0);
- зелёный контур: можно стрелять (разведана и живая).

Приоритет: оранжевый не перекрашивается зелёным (если клетка заблокирована, остаётся оранжевой).

На поле игрока дополнительная подсветка:
- тёмно-серый контур используется для обозначения клеток игрока, которые соперник ещё не разведал.

Перемещение:
- белое свечение контура показывается для клеток, куда можно переместить выбранный набор ЮнитА (валидные по правилам перемещения).

---

## 14) Турбо-битва
Если после завершения раунда 7 у обоих игроков остались ЮнитА:
- начинается турбо-битва до победы;
- таймер хода 15 секунд;
- фазы те же: действия → ЮнитР;
- у игрока 1 ЮнитР, который автоматически создаёт блокировку и разведку выбранной клетки;
- урон атак умножается на 3.

---

## 15) Победа
Победа наступает, когда у соперника не осталось ЮнитА на поле.

---

## 16) Матчмейкинг PvP (до 0 раунда)

### 16.1 Экран поиска
После нажатия “Играть” появляется чёрный экран. По центру:
- квадрат размером с игровую ячейку;
- заливка чёрная, контур белый;
- от контура идут “волны по воде” (анимация).

### 16.2 Алгоритм поиска
Сервер ищет соперника среди активных игроков в очереди.

**Максимальное время поиска: 3 минуты (180 секунд).**

Диапазон рейтинга:
- старт: ±1;
- каждые 0.1 секунды расширение диапазона на +1;
- на каждом расширении делается запрос на сервер на поиск соперника в текущем диапазоне.

Дополнительно: **лесенка расширения радиуса поиска** (ограничитель/корректор диапазона):
- после 60 секунд ожидания диапазон принудительно становится не меньше **±5**;
- после 120 секунд ожидания диапазон принудительно становится не меньше **±10**.

> Примечание для разработки: “лесенка” трактуется как минимальный гарантированный диапазон.  
> То есть фактический диапазон на момент запроса = `max(текущий_диапазон_по_0.1с, лестничный_минимум)`.

Если за 180 секунд соперник не найден — поиск завершается (поведение на клиенте: возвращение в меню/сообщение — определить отдельно).

### 16.3 Блок соперника на 60 минут
После завершения матча между двумя игроками ставится блок на повторную игру длительностью 60 минут.
При подборе пары проверяется блок; если блок активен — пара запрещена.

### 16.4 Матч найден: стартовая анимация
Когда соперник найден:
- внутри центрального квадрата начинается отсчёт 5 → 0;
- над квадратом показывается карточка соперника: картинка/аватар, ник, рейтинг;
- на фоне всплывает поле (ячейки как будто поднимаются из-под воды);
- карточка соперника остаётся видимой до “0”;
- на “0” начинается раунд 0.

---

## 17) Игровое окно: счёт и индикаторы
В верхней панели отображается счёт в виде:

`xxxxx 25 : 25 xxxxx`

- слева 5 иконок ЮнитА игрока, справа 5 иконок ЮнитА соперника;
- числа — суммарные HP текущих ЮнитА игрока и соперника;
- при уничтожении ЮнитА контур “крайнего от центра” (дальнего от счёта) значка становится красным, по одному за каждую смерть.

---

## 18) Режим “Тренироваться” (против компьютера, локально)

### 18.1 Запуск тренировки
В главном меню две кнопки: “Играть” и “Тренироваться”.

При нажатии “Тренироваться”:
- сразу запускается отсчёт 5 → 0 (как при нахождении соперника);
- вместо карточки игрока показывается карточка компьютера:
  - ник: Qubiq
  - рейтинг (уровень): 500
  - картинка: подгружается (если не успела — плейсхолдер)
- на “0” начинается раунд 0.

### 18.2 Единственный бот (Qubiq, простая сложность)
Бот работает строго по тем же правилам, что и игрок, и использует только доступную информацию (разведанные клетки).

Раунд 0:
- бот расставляет ЮнитА случайными порциями по случайным клеткам своего сектора, пока не выставит все 5;
- бот расставляет ЮнитР по полю игрока случайно: выбирает клетку и ставит 1 или 2 (с учётом лимита 2 и остатка), пока не распределит все 4.

Раунды 1+:
Фаза действий:
- бот случайно выбирает для своих доступных ЮнитА действие (атака/перемещение/ничего) с учётом валидности.
- при атаке бот оперирует пакетами: может атаковать всем набором в клетке или выбрать подмножество, может ошибаться и разбивать пакет.

Если бот видит разведанную клетку игрока с ЮнитА и выбирает атаку:
- выбирает “целевую” клетку с ЮнитА игрока (приоритет: максимальная суммарная HP в клетке, если вариантов больше двух);
- затем выбирает фактическую цель атаки как случайную клетку из области 3×3 вокруг целевой (1 из 9, включая целевую);
- если выбранная цель невалидна для атаки, бот перебирает вариант внутри этих 9 до валидного; если валидных нет — переходит к перемещению/ничего по правилам ниже.

Фаза ЮнитР:
- если бот не видит ни одной клетки игрока с ЮнитА, отправляет все 4 ЮнитР в туман войны (координатно “вслепую”), соблюдая лимит 2 на клетку;
- если видит клетки с ЮнитА, случайно выбирает между разведкой и блокировкой (если хватает ЮнитР);
- блокировку ставит в области 3×3 вокруг выбранной “целевой” клетки (как 1 из 9), может повторять, пока есть ЮнитР и есть возможность поставить блокировку;
- если видимых целей больше двух, выбирает клетку с максимальной суммой HP ЮнитА.

Отдельное правило “если бот не видит возможности для атаки”:
- если у бота нет валидных целей для атаки, он случайно выбирает: перемещать или нет ЮнитА;
- если перемещение невозможно, а атака возможна, он стреляет;
- если ни атака, ни перемещение невозможны, он ничего не делает.

## 19) Профиль игрока и идентификация устройства

### 19.1 Идентификатор
Идентификатором игрока служит сгенерированный UUID при первом запуске клиента, сохранённый локально. Это используется как “аккаунт устройства” без привязки к Google Play.

На первом запуске/первом входе:
- игрок выбирает аватар из нескольких;
- игрок вводит ник.

Требования к нику:
- допускаются заглавные/строчные латинские буквы, кириллица и цифры;
- **длина ника меньше или равна 15 символам**;
- позже ник и аватар можно будет менять.

## 20) Рейтинг (внутренний, ELO-подобный)

Пока хранится только рейтинг, без истории матчей.

Начальный рейтинг нового игрока: **1000**.

Базовый коэффициент K принят равным 10.

Ожидаемая вероятность победы игрока A:
`E_A = 1 / (1 + 10^((R_B - R_A)/400))`

Фактический результат:
`S_A = 1` при победе A, иначе `0`

Изменение рейтинга:
`ΔA = round_math(10 * (S_A - E_A))`

Округление:
- математическое округление до целого;
- важно, чтобы изменение у победителя и проигравшего было одинаковым по модулю и противоположным по знаку.

Новый рейтинг:
`R_A' = R_A + ΔA`
`R_B' = R_B - ΔA`

## 21) PvP: обработка разрывов соединения

Игра синхронная, раунды доигрываются с тем, что успели назначить. При разрывах соединения действуют правила технического поражения.

Определения:
- “дисконнект” — потеря соединения игрока с сервером (WebSocket/heartbeat).

Правило для первого дисконнекта за матч:
- при первом дисконнекте ждём 30 секунд;
- вне зависимости восстановилось соединение или нет, раунд доигрывается и резолвится;
- если снова дисконнект, ждём ещё 30 секунд, запускаем следующий раунд по таймеру и резолвим;
- если после этого соединения нет — техническое поражение.

Правило для второго и последующих дисконнектов за матч:
- ждём 30 секунд один раз;
- доигрываем раунд и резолвим;
- если соединения нет — техническое поражение.

## 22) Технологический стек (рекомендуемый под текущую концепцию)

Проект задуман так, чтобы его можно было вести в одиночку, писать код через Codex, хранить версии в git и поддерживать сервер.

### 22.1 Клиент
Рекомендуемый движок: Godot 4.x (2D).

Тренировка (бот) реализуется полностью локально на клиенте.

Сеть для PvP:
- HTTP для регистрации/профиля/инициации матчмейкинга;
- WebSocket для статусов матча, событий резолва и обмена командами.

### 22.2 Сервер
Rust как основной язык.

Рекомендуемые компоненты:
- Axum для HTTP API;
- WebSocket (через подходящий слой в Axum/Tokio) для матчей и событий;
- Tokio как runtime.

Хранилища:
- PostgreSQL для профиля пользователя и рейтинга, а также блоков соперников на 60 минут;
- Redis для очереди матчмейкинга, transient-статусов и быстрых TTL-ключей.

Деплой:
- Docker Compose на VPS.
- Reverse proxy (Nginx/Traefik) для TLS (HTTPS/WSS).

Наблюдаемость:
- структурированные логи;
- минимальные метрики (онлайн, матчей/мин, средний поиск, ошибки).

---

## 23) Архитектурные принципы симуляции (важно для корректности)
- В PvP сервер всегда авторитетен и единственный источник истины.
- Клиент отправляет команды (пакеты действий/ЮнитР), сервер считает резолв.
- Резолв должен быть детерминированным при одинаковых входных данных.
- Порядок резолва фиксирован: перемещения → атаки (с суммированием) → применение ЮнитР.
